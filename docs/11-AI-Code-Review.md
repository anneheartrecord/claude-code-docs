# 11 Code Review in the AI Coding Era

## 1️⃣ It Started with a Sourcemap Leak

In 2025, Claude Code's complete source code was publicly reconstructed. The reason was almost laughable: **the build process failed to exclude sourcemap files.** Sourcemaps are debug-aid files generated by JavaScript build tools that record the mapping between compiled code and original source. With .map files in hand, the original code can be reconstructed almost perfectly.

This project has 515,000 lines of code. TypeScript type definitions have zero errors. Every async function has proper error handling. The code structure is clean and well-organized. From a code quality perspective, it's nearly impeccable. Yet on the most fundamental release security question of "what files should the build output contain," there was a fatal oversight.

This incident perfectly reveals a core contradiction of the AI Coding era: **AI is getting increasingly stronger at the code-writing level, but still has blind spots at the engineering judgment level.** AI can write flawless code, but it may not be able to judge whether the build artifacts of that code should be made public.

**This is precisely why Code Review needs to be redefined in the AI era.**

## 2️⃣ Individual Level: Shifting the Review Focus

### The Old Approach

Before AI Coding became widespread, Code Review was one of the most important quality assurance practices in software engineering. Reviewers focused on whether code style was standard, whether variable names were clear, whether design patterns were appropriate, and whether edge cases were covered. A thorough CR could take anywhere from 30 minutes to several hours.

In early 2024, when many developers first started using AI to write code, they would carefully review every line of AI-generated output. This was a cautious attitude, entirely reasonable when AI capabilities were still immature.

### What Has Changed

As model capabilities have improved rapidly, line-by-line review is becoming uneconomical. **If the purpose of using AI is to boost productivity, but the time spent reviewing AI code is roughly the same as writing it yourself, then AI's efficiency advantage is severely diminished.**

Between 2025 and 2026, more and more engineers have adjusted their CR strategies in practice. The core trend is: **shift attention from what AI does well to what AI does poorly.**

Areas where AI already performs well enough:

- **Code style and naming conventions.** AI-generated code is typically more consistent and standardized than most human engineers' output. Correcting AI's naming style is almost a waste of time.
- **Type annotations and interface definitions.** Taking TypeScript as an example, AI almost never misses type annotations, and interface definitions are usually complete.
- **Error handling coverage.** AI tends to add try/catch or error checks for every place that could potentially fail.

Areas where AI still has blind spots, and where human review should focus:

- **Business logic correctness.** AI writes code based on the prompt, but the prompt itself may be ambiguous or may omit certain business rules. Whether the logic is correct can only be judged by someone who understands the business.
- **Security.** AI doesn't proactively consider injection attacks, privilege escalation, or information leakage. It can write functionally correct code, but not necessarily secure code.
- **Architectural decisions.** Whether a certain module should be split, whether a certain dependency should be introduced, whether a certain API should be exposed externally. AI provides locally optimal solutions; global architectural judgment requires a human.

**The goal of CR is shifting from "finding code problems" to "verifying whether AI's decisions match expectations."**

## 3️⃣ Team Level: Shared System Prompt Baseline

If every member of a development team uses their own prompt to guide AI in writing code, the resulting merged codebase will have inconsistent styles and architectures. Each person's AI might use different error handling approaches, different naming conventions, and different testing strategies.

The solution is to establish a **team-shared System Prompt baseline**. In Claude Code, this is the CLAUDE.md file. Other AI coding tools have equivalent configuration mechanisms.

This shared prompt should clearly define:

**Tech stack and architecture conventions.** What frameworks the project uses, what language versions, what build tools. How modules depend on each other. How data flows through the system.

**Code style and naming conventions.** Whether variables use camelCase or snake_case, what format file names follow, how components are organized.

**Commit conventions and branching strategy.** The format for commit messages, branch naming rules, and PR merge strategies.

**Testing requirements.** Which code must have unit tests, what the minimum coverage requirements are, and how to write integration tests.

**Security red lines.** Which operations must never appear in code, such as hardcoded keys, plaintext password storage, or unrestricted file access.

This is fundamentally the same thing as a Coding Standard document in a traditional team. The difference is: the audience used to be human engineers, and now it's AI. And AI is better than humans at strictly following specifications — provided those specifications are clearly expressed in the prompt.

**Without a shared prompt, the code style, architecture style, and testing style from each person's AI will all be different.** The codebase eventually becomes a hodgepodge. With a shared prompt, at least a consistent baseline is guaranteed.

## 4️⃣ CICD Infrastructure Level: A New Paradigm for the AI Era

This is the layer that most needs to be redefined, and also the most valuable.

### Traditional CICD Pass Criteria

Traditional CICD pipelines focus on: whether code compiles, whether it runs correctly, and whether all tests pass. These checks assume code is manually written by experienced engineers, with problems mainly stemming from logic errors and oversights.

### Checks CICD Needs to Add in the AI Era

When the primary code producer shifts from humans to AI, the CICD pipeline needs a series of additional checks targeting AI code characteristics.

**1️⃣ Sensitive Information Scanning**

When AI generates code, it may hardcode sensitive information mentioned in the prompt — API keys, internal URLs, database connection strings — directly into the code. The CI pipeline should automatically scan for these patterns, and any hardcoded credentials should trigger a build failure.

Commonly used tools include git-secrets, truffleHog, and gitleaks. These tools existed before the AI era but are now more necessary than ever.

**2️⃣ Privacy Checks**

AI sometimes leaks prompt information into code comments, log output, or error messages. For example, if you describe an internal system's architecture in the prompt, AI might write that description as a comment in the code. The CI pipeline should scan for internal information appearing where it shouldn't.

**3️⃣ Build Artifact Auditing**

The Claude Code sourcemap leak was caused by the absence of this exact check. The CI pipeline should verify that final build artifacts don't contain:

- .map files, i.e., sourcemaps
- Debug symbol tables
- Internal documentation or comments
- Test data and fixtures
- Environment configuration files

**4️⃣ Dependency Security Auditing**

AI tends to introduce third-party libraries to solve problems. Every new dependency is a potential supply chain attack surface. The CI pipeline must audit dependencies for security.

In the JavaScript/Node.js ecosystem, `npm audit` is the most basic tool. More professional tools like Snyk and Socket.dev can perform deeper analysis, including detecting malicious packages and license compliance.

**5️⃣ Code Review Agent**

This is the most promising direction: **introduce a Review Agent that performs automated review on every commit.**

The specific approach is to commit the requirement prompt alongside the code implementation. The Review Agent reads the prompt and checks whether the code matches expectations. This creates an **AI-checks-AI** feedback loop.

```
Commit contents:
├── prompt.md              # Original requirement prompt
├── src/feature.ts         # AI-generated code
└── tests/feature.test.ts  # AI-generated tests

Review Agent checklist:
1. Does the code implementation match requirements described in prompt.md?
2. Are there security vulnerabilities? Injection, privilege escalation, information leakage
3. Is there hardcoded sensitive information?
4. What scenarios do the tests cover? What edge cases are missing?
5. Are there unnecessary dependencies introduced?
6. Does the code contain prompt information leakage?
```

**A new CICD paradigm for the AI era should be formally established as part of Harness Engineering — perhaps even the most important part.** When the code producer shifts from humans to AI, traditional manual review is no longer sufficient. You need infrastructure to systematically guarantee quality.

## 5️⃣ An Actionable AI Code Review Workflow

Integrating all the layers above into a complete workflow:

```
Step 1: AI generates code
  Original prompt submitted alongside
     ↓
Step 2: Automated checks via CI pipeline
  - lint / type check / build
  - Sensitive information scanning
  - Build artifact auditing, sourcemap / debug file detection
  - Dependency security auditing
     ↓
Step 3: Review Agent — AI reviews AI
  - Requirement conformance check
  - Security audit
  - Edge case coverage
  - Prompt information leakage check
     ↓
Step 4: Human review — critical paths only
  - Business logic correctness
  - Architectural decision soundness
  - Security attack surface assessment
     ↓
Step 5: Merge and release
```

The core philosophy of this workflow is **layered filtering**. Automated checks filter out the most basic issues, the Review Agent filters out AI-specific problems, and human review only needs to address the highest-level judgments. Each layer does what it does best.

## 6️⃣ Review Lessons from Claude Code's Source

Claude Code's 515,000-line source is an excellent case study for AI code review. By analyzing its code characteristics, we can distill a review guide specifically for AI-generated code:

| Typical AI Code Characteristic | What to Focus On |
|-------------------------------|-----------------|
| Every async function has try/catch | Does the catch block just log and swallow the error? Swallowed errors hide problems and make debugging extremely difficult |
| Extremely thorough type annotations | Are there `as any` casts bypassing the type system? Each `as any` is a type safety escape hatch worth scrutinizing |
| All branch conditions are handled | Are else branches just placeholders that do nothing? AI likes writing "complete" code, but the else might just return undefined |
| Error messages are comprehensive | Do error messages leak internal implementation details? Such as file paths, database table names, or internal API addresses |
| Large codebase with neat structure | Is there massive duplicated code that could be abstracted? AI tends to copy-paste similar logic rather than extracting common functions |
| Many dependencies introduced | Are there unnecessary third-party libraries? AI might import an entire library for a simple function |
| Build passes, all tests green | Do build artifacts contain files that shouldn't be there? This is exactly the lesson from Claude Code's sourcemap leak |

## 7️⃣ Core Principle

**AI's efficiency advantage in writing code lies in generation speed.** It can write hundreds of lines of fully functional code in seconds — something humans simply cannot do.

**The irreplaceable value of human involvement lies in security judgment and engineering decisions.** AI sees the current task; humans see the entire system. AI optimizes for functional correctness; humans need to ensure security, maintainability, and architectural soundness.

**CICD infrastructure is the Harness that ties both together.** It encodes human engineering experience into automated rules, applying them continuously to every code commit.

In the AI Coding era, **the biggest danger is not AI writing buggy code, but teams relaxing engineering standards around AI code.** Code production efficiency has increased tenfold; quality assurance investment should increase correspondingly — but the approach needs to shift from human-intensive to infrastructure-intensive.
