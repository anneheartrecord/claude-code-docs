# 04 上下文工程

## 核心问题

上下文窗口是有限的。你必须在有限的空间里塞进最有价值的信息，同时保证模型的注意力不被稀释。

Claude Code 在这个问题上做了一套完整的分层优先级 + 预取缓存 + 动态注入体系。

## System Prompt 构建

分两个阶段动态构建：

### 阶段一：静态部分
- 内置 Agent 行为规则
- 可用的 slash command 列表
- 模型配置、thinking 参数
- 权限模式描述

### 阶段二：动态部分

**Git 状态（context.ts）：**
- 并行执行 4 个 git 命令：当前分支、status --short、最近 commit、用户名
- 结果限制在 2000 字符内，超出截断并附带警告
- session 内 memoize 缓存，只取一次
- 远程模式（CCR）或用户禁用时跳过

**CLAUDE.md 记忆（claudemd.ts，1,400 行）：**
- 五层优先级加载（低到高）：
  1. 全局管理级 `/etc/claude-code/CLAUDE.md`
  2. 用户级 `~/.claude/CLAUDE.md`
  3. 项目级 `CLAUDE.md`、`.claude/CLAUDE.md`、`.claude/rules/*.md`
  4. 本地级 `CLAUDE.local.md`（gitignore，私有）
  5. 自动记忆（feature flag: TEAMMEM）

- **@include 指令系统：**
  - 语法：`@path`、`@./relative`、`@~/home`、`@/absolute`
  - 只在叶子文本节点中解析，不会在代码块里触发
  - 循环引用检测，通过 Set 追踪已处理文件
  - 支持 200+ 文件扩展名，自动屏蔽二进制文件

- **关键常量：**
  - `MAX_MEMORY_CHARACTER_COUNT = 40000`（总字符上限）
  - `MAX_ENTRYPOINT_LINES = 200`（MEMORY.md 行数上限）
  - `MAX_ENTRYPOINT_BYTES = 25000`（MEMORY.md 字节上限）

**权限规则：**
- 从 `~/.claude/settings.json` 和 `.claude-permissions.json` 加载
- 解析为 ToolPermissionContext，包含 allow/deny/ask 规则

## 分层优先级

当上下文接近上限时，各部分被保留的优先级：

```
优先级从高到低：

1. 系统指令（内置行为规则）          ← 永不压缩
2. CLAUDE.md 用户记忆               ← 永不压缩
3. 权限规则                         ← 永不压缩
4. 最近的用户消息和 assistant 回复   ← 最后压缩
5. 历史工具调用结果                  ← 最先被微压缩清理
6. 早期对话历史                     ← Full Compact 时被摘要替换
```

这个优先级保证了即使上下文被压缩，用户的个性化配置和行为规则永远不会丢失。

## 预取与缓存

避免每次循环都做 IO 是性能的关键：

| 信息 | 预取策略 | 缓存策略 |
|------|---------|---------|
| Git 状态 | query loop 开始时异步预取 | session 内 memoize，只取一次 |
| CLAUDE.md | 启动时异步加载 | session 内缓存，不重新读 |
| Skill 列表 | 后台 discovery | 文件监听，变化时热更新 |
| 工具定义 | 启动时注册 | feature flag 控制，运行时不变 |

## 动态上下文注入

不同阶段注入不同的上下文，避免所有信息一股脑塞进去：

- **Planning 阶段：** 注入全量 Skill 列表和项目结构
- **执行阶段：** 只注入当前任务相关的文件内容
- **压缩后恢复阶段：** 注入摘要 + transcript 文件路径（按需读取）

## Prompt Cache 优化

Claude Code 对 prompt cache 做了精细的管理：

**CacheSafeParams：** 父子 Agent 之间共享一组参数，包含 systemPrompt、userContext、systemContext、toolUseContext。只有这些参数完全匹配时，API 请求才能命中 prompt cache。

**cache break detection：** 专门的检测模块监控 cache 命中率。如果突然下降，区分是真正的 break 还是压缩/微压缩导致的正常下降。

生产教训（BQ 2026-03-22）：77% 的工具 cache break 是因为 AgentTool 和 SkillTool 的 description 里嵌入了动态列表，每次列表变化都会导致 cache 失效。
