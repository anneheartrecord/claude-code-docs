# 11 AI Coding 时代的 Code Review

## 从 Sourcemap 泄露说起

Claude Code 的源码泄露本身就是一个绝佳的案例。一个 AI 写的 51.5 万行项目，构建时忘了排除 sourcemap 文件，导致完整源码被还原。

AI 把功能写得很好，类型零错误，但在发布安全这种工程化细节上留了一个致命的漏洞。

**这恰恰说明了 AI Coding 时代，Code Review 的重心需要彻底转移。**

## 三个层面的 Review 范式

### 1️⃣ 个人层面：从精细 CR 到只看关键路径

去年我还会对 AI 写的每一行代码做比较详细的 CR，保证可读性、命名规范、设计模式。但随着模型能力的提升，以及用 AI 本身就是为了速度，如果在 CR 上花太多时间，这个工具就丧失了大部分价值。

**今年我几乎已经不做传统意义上的 CR 了。** 核心接口可能会看一下逻辑的正确性，其他的直接放过。

这个转变的底层逻辑是：AI 在代码风格、命名规范、设计模式这些"形式"层面已经做得足够好了，不需要人去纠。人的注意力应该放在 AI 做不好的地方：

- **业务逻辑的正确性：** AI 不懂你的业务，它只是按 prompt 写代码。逻辑对不对只有你知道
- **安全性：** AI 不会主动考虑注入、越权、信息泄露。这些你得自己盯
- **架构决策：** 这个模块该不该拆、这个依赖该不该引入、这个接口该不该暴露。AI 给的是局部最优解，全局判断靠人

大部分工程师都应该适应这种新的 CR 模式。**CR 的目的从"找代码问题"变成了"验证 AI 的判断是否符合预期"。**

### 2️⃣ 团队层面：共享 System Prompt 基准

一个开发团队应该有一份公用的基准 system prompt，对这个项目的技术规范有着精准的定义：

- 项目的技术栈和架构约定
- 代码风格和命名规范
- 提交规范和分支策略
- 测试要求
- 安全红线

这份 system prompt 就是 CLAUDE.md 或等效的配置文件。它的作用是保证团队每个成员 AI Coding 的产物不会跑偏太多。

**没有共享 prompt 的团队，每个人的 AI 写出来的代码风格、架构风格、测试风格都不一样。** 合到一起就是一锅乱炖。有了共享 prompt，至少保证了一个基准线。

这和传统团队里的 coding standard 文档是一个道理，只不过受众从人变成了 AI。

### 3️⃣ CICD 基础设施层面：AI 时代的新范式

这是最需要被重新定义的一层。

**传统 CICD 的通过标准：** 可编译、可运行、测试通过。

**AI 时代 CICD 应该增加的标准：**

**安全扫描。** 扫描 AI 生成的代码中有没有硬编码的 API Key、内部 URL、调试配置。Claude Code 的 sourcemap 泄露就是因为发布流程里没有这一步。

**隐私检查。** AI 有时候会把 prompt 中的信息泄露到代码注释或日志里。CI 里应该扫描有没有敏感信息出现在不该出现的地方。

**Sourcemap / Debug 文件检测。** 构建产物里有没有不该出现的 .map 文件、debug 符号表、内部文档。

**依赖安全审计。** AI 喜欢引入第三方库来解决问题，每个新依赖都是一个供应链攻击面。CI 里跑 npm audit / snyk 是必须的。

**Code Review Agent。** 这是最有意思的一个方向。引入一个 Review Agent，对每一次提交做完整的审查。

具体做法：把需求提示词和代码实现一起 commit。Review Agent 读取提示词，看代码实现是否符合预期。用 AI 来 Review AI。

```
提交内容：
├── prompt.md          # 原始需求提示词
├── src/feature.ts     # AI 生成的代码
└── tests/feature.test.ts  # AI 生成的测试

Review Agent 的检查清单：
1. 代码实现是否符合 prompt.md 中描述的需求？
2. 有没有安全漏洞？（注入、越权、信息泄露）
3. 有没有硬编码的敏感信息？
4. 测试覆盖了哪些场景？遗漏了哪些边界条件？
5. 有没有引入不必要的依赖？
```

**AI 时代 CICD 的新范式应该被确立，作为 Harness Engineering 的一部分，甚至是最重要的一部分。** 因为当代码的生产者从人变成了 AI，传统的人工 Review 就不够用了。你需要用基础设施来保证质量。

## 可落地的 AI 代码 Review 流程

```
1. AI 生成代码（附带原始 prompt）
     ↓
2. 自动化检查（CI 流水线）
   - lint / type check / build
   - 敏感信息扫描
   - sourcemap / debug 文件检测
   - 依赖安全审计
     ↓
3. Review Agent（AI Review AI）
   - 需求符合度检查
   - 安全审计
   - 边界条件覆盖
     ↓
4. 人工 Review（只看关键路径）
   - 业务逻辑正确性
   - 架构决策合理性
   - 安全攻击面
     ↓
5. 合并发布
```

## 从 Claude Code 源码学到的 Review 教训

| AI 代码特征 | 该重点看什么 |
|------------|-------------|
| 每个 async 都有 try/catch | catch 里是不是只做了 log 就吞掉了错误 |
| 类型标注极其完整 | 有没有用 as any 强行绕过类型检查 |
| 所有分支都有处理 | else 分支是不是只是占位符 |
| 错误信息很完整 | 有没有泄露内部实现细节 |
| 代码量大但结构整齐 | 有没有大量重复代码可以抽象 |
| 依赖多 | 有没有引入不必要的库 |
| 构建通过 | 构建产物里有没有不该出现的文件 |

**核心原则：AI 写代码的效率优势在于生成速度，人工的不可替代价值在于安全判断和工程化细节。CICD 基础设施则是把这两者串联起来的 Harness。**
