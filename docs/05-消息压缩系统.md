# 05 消息压缩系统

## 为什么需要压缩

Agent 的对话越长，消息历史越大。一个复杂任务可能涉及几十次工具调用，每次 FileRead 可能返回几千 token 的文件内容。不做压缩，几轮对话就撑爆上下文窗口。

Claude Code 的压缩系统位于 `src/services/compact/`，包含 26 个文件，是整个代码库中工程化程度最高的模块之一。

## 三层压缩体系

```
每次 API 调用前
  ↓
第一层：微压缩（Microcompact）
  - 清理旧的工具调用结果
  - 两种方式：时间触发 / 缓存编辑
  ↓
检查 token 用量 > 上下文窗口 - 13K？
  ↓ 超过
第二层：Session Memory Compact（优先尝试）
  - 关键信息沉淀到记忆文件
  - 删除已沉淀的旧消息
  ↓ 不够
第三层：Full Compact
  - Fork 独立 Agent 做全量摘要
  - 九段式结构化模板
  - 替换全部历史为一条摘要消息
```

## 第一层：微压缩

### 触发条件

每次 API 调用前自动执行，不需要达到任何阈值。

### 处理范围

只压缩特定工具的输出，这些工具产出大但时效性差：
- FileRead、FileEdit、FileWrite
- Bash（所有 shell 工具）
- Grep、Glob
- WebSearch、WebFetch

用户消息、assistant 文本、thinking block 永远不动。

### 两种实现方式

**时间触发型（Time-based MC）：**

如果距离上一次 assistant 消息超过配置的时间阈值，说明服务端 prompt cache 已过期。此时直接把旧的 tool_result 内容替换为 `[Old tool result content cleared]`，保留最近 N 个工具结果不动。

```
触发条件：距上次 assistant 消息的分钟数 > gapThresholdMinutes
保留策略：保留最近 keepRecent 个（最少保留 1 个）
做了什么：tool_result.content = "[Old tool result content cleared]"
```

**缓存编辑型（Cached MC，feature flag: CACHED_MICROCOMPACT）：**

不修改本地消息内容，而是通过 API 层的 `cache_edits` 机制告诉服务端删除指定 tool_result 的缓存。这样不破坏已有的 prompt cache，性能更好。

```
触发条件：已注册的 tool_result 数量 > triggerThreshold
保留策略：保留最近 keepRecent 个
做了什么：生成 cache_edits block 发给 API，本地消息不变
```

这两种方式互斥。缓存编辑型优先，如果不支持（模型不支持或外部构建），fallback 到时间触发型。

## 第二层：Session Memory Compact

### 触发条件

自动压缩触发时优先尝试。如果 Session Memory 能释放足够空间，就不需要做 Full Compact。

### 做了什么

把会话中的关键信息抽取出来写入持久化的 session memory 文件，然后从消息历史中删除已经被记录的旧消息。相当于把消息历史"沉淀"到磁盘，为上下文腾出空间。

## 第三层：Full Compact

### 触发条件

token 用量超过上下文窗口 - 13,000 的缓冲区，且 Session Memory 压缩不够用。

### 关键常量

```
AUTOCOMPACT_BUFFER_TOKENS = 13,000    // 触发缓冲区
WARNING_THRESHOLD_BUFFER_TOKENS = 20,000  // 警告阈值
MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20,000    // 摘要最大输出
MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3   // 熔断器阈值
```

### 执行过程

Fork 一个独立的 Agent 做压缩。不在主循环里做，避免阻塞。

压缩 Agent 收到整个对话历史，按九段式模板生成摘要：

```
1. 用户的主要请求和意图
2. 关键技术概念
3. 涉及的文件和代码片段（带完整代码）
4. 遇到的错误和修复方式
5. 问题解决过程
6. 所有用户消息原文（非工具结果的）
7. 待完成的任务
8. 当前正在进行的工作
9. 下一步计划
```

### analysis + summary 两阶段生成

压缩 prompt 要求 LLM 先在 `<analysis>` 标签里做分析思考，再在 `<summary>` 标签里写正式摘要。最终只保留 summary 部分注入上下文，analysis 被丢弃。

这是一个提升摘要质量的 prompt 技巧。让 LLM 先想清楚再总结，但不浪费上下文空间。

### 两种压缩方向

- **全量压缩（BASE）：** 摘要整个对话
- **部分压缩（PARTIAL）：** 只摘要最近的消息，保留更早期的已摘要内容。分 `from`（从某条消息开始）和 `up_to`（到某条消息为止）两个方向

### 压缩后处理

1. 旧消息全部删除，替换为一条包含摘要的 user message
2. 完整对话写入 transcript 文件做备份
3. 重置微压缩状态、Session Memory 指针、上下文折叠状态
4. 通知 prompt cache detection：这次 cache 命中率下降是压缩导致的，不要报假警

## 工程化细节

### 熔断器（Circuit Breaker）

如果自动压缩连续失败 3 次，停止重试。

代码注释（BQ 2026-03-10）：1,279 个 session 出现了 50 次以上的连续压缩失败，最高 3,272 次，每天浪费约 25 万次 API 调用。加了熔断器之后这个问题消失了。

### 递归保护

压缩本身也是通过 LLM 调用完成的。如果压缩 Agent 的上下文也超了怎么办？代码里做了显式的递归守护：

```typescript
if (querySource === 'session_memory' || querySource === 'compact') {
  return false  // 不再触发自动压缩
}
```

同理，Context Collapse 模式下也不触发自动压缩，两者会互相竞争。

### Prompt Cache 感知

压缩会破坏已有的 prompt cache。代码里专门有 `notifyCompaction` 机制，告诉 cache break detection 模块：这次 cache 命中率下降是压缩导致的。

生产教训（BQ 2026-03-01）：Session Memory 压缩后缺少 cache baseline 重置，导致 20% 的 cache break 事件是误报。
